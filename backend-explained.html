TL;DR in one paragraph

The backend is a single Go binary that listens on one HTTP port. On startup it reads env vars, opens a SQLite database, connects to S3/SES/Stripe/Twilio, wires all feature handlers into a router, and starts serving. Requests flow through a chain of middleware (logging, recovery, CORS, auth, rate limiting) before reaching a handler. There are two auth systems: a cookie-based session for providers who log in, and signed magic-link tokens for parents/staff/signers who click email links. Schema changes live in numbered SQL files under migrations/ and are applied by the migrate CLI.

1. The big picture

Your backend is a monolith — one program, one binary, one process. That's a good choice for a bootstrapped SaaS: everything is in one place, deployment is one scp or docker push, and you can reason about the whole system at once. When you're bigger, you split it. Not before.

Here is what the process looks like from the outside:

                 ┌────────────── Browser / mobile / webhook ─────────────┐
                 ▼                                                       ▼
            HTTPS (80/443)                                        HTTPS (Stripe)
                 │                                                       │
                 ▼                                                       ▼
        ┌─────────────────────────────────────────────────────────────────┐
        │                  compliancekit-api  (one Go binary)            │
        │                                                                 │
        │   router → middleware chain → handler → db / S3 / Stripe / …   │
        └─────────────────────────────────────────────────────────────────┘
              │           │             │             │           │
              ▼           ▼             ▼             ▼           ▼
           SQLite        AWS S3       AWS SES      Stripe       Twilio
         (local file)  (docs)       (email)    (subscriptions)  (SMS)

Requests come in on one port. The binary talks to one local SQLite file for state, and to a handful of external APIs (S3 for document storage, SES for email, Stripe for billing, Twilio for SMS, Mistral/Gemini for OCR). That's it. There are no microservices, no queues, no Kubernetes, no message bus. Keep it that way as long as you can.

2. Shape of a Go project (quick orientation)

If you're new to Go, the layout looks weird at first. Here's the translation:

FolderWhat it means

internal/{billing, storage, notify, magiclink, ocr, pdfsign, compliance, immunization}Business logic for each subsystem. Each is a Go package.
migrations/Versioned SQL files. 000001_init…up.sql is the forward migration, …down.sql is the rollback.

MakefileShortcuts: make run, make build, make migrate-up, etc.
Dockerfile, docker-compose.ymlContainer image + local dev stack.
Go mental model: a package is a folder. A module (declared in go.mod) is the whole project. Imports are by URL-ish path, e.g. github.com/markdonahue100/compliancekit/backend/internal/db. The compiler refuses to compile unused imports or unused variables — that's a feature, not a bug.

3. Startup: what happens when the server boots

All of this lives in cmd/server/main.go. When you run make run or the compiled binary, the sequence is:

  1. Load config — config.Load() reads every env var into a typed struct. If anything ,required is missing, the process exits immediately with an error. You never get a half-configured server.
  2. Build the logger — slog (the Go standard library structured logger) emits JSON. Every log line has app=compliancekit-api env=production level=info …. Made the default so every package logs the same way.
  3. Set up signal handling — signal.NotifyContext makes a context that cancels on Ctrl-C (SIGINT) or docker stop (SIGTERM). That context is passed to everything — when it cancels, the whole app shuts down cleanly.
  4. Open the database — db.Open opens SQLite with the right pragmas (WAL mode, foreign keys on, 5s busy timeout). It pings to confirm it works, then returns a pool.
  5. Build the external clients — one AWS S3 client (documents), one SES client (email), one Twilio SMS sender, one Stripe billing service. If SES fails to init, we log a warning and keep running without email (graceful degradation).
  6. Instantiate the magic-link service with the HMAC signing key from config.
  7. Pick an OCR engine — Mistral primary, Gemini fallback. If neither key is set, OCR is disabled but uploads still work.
  8. Create handler structs — each handler (ProviderHandler, ChildHandler, etc.) is a plain struct with pointers to the db pool, storage, logger, etc. This is Go's version of dependency injection: no framework, just fields on a struct.
  9. Start background workers — go chase.RunDaily(rootCtx) launches the daily notification chase in a goroutine (a lightweight thread). It stops when rootCtx cancels.
  10. Build the router — api.NewRouter wires every URL to a handler method. More on this below.
  11. Start listening — http.Server starts accepting connections on the configured port. Timeouts are set so a slow client can't hog a goroutine forever.
  12. Block until signal — main waits on either a server error or the shutdown signal. On shutdown it calls srv.Shutdown(ctx) which stops accepting new connections and drains in-flight ones (up to 20s), then exits.

The important thing: there is no magic lifecycle. main() is ~150 lines that do every step in order. When something breaks at startup, you can read top-to-bottom and find it.

4. The HTTP server & router

Go's standard library ships with net/http, which already knows how to listen on a port and dispatch requests. We add one small dependency — chi — for routing because the standard library's router is intentionally minimal (no path params, no sub-routers).

What's a router?

A router maps (method, path) pairs to functions. r.Get("/api/children", children.List) says: "when a GET request arrives at /api/children, call children.List." That's it.

What's middleware?

Middleware is a function that wraps a handler. It runs before (or after) the handler, sees the same request, can modify it, short-circuit it, or let it pass. You stack middleware like onion layers: request enters from the outside, flows through each layer, hits the handler, flows back out.

In internal/api/router.go the top-level chain is:

r.Use(chimw.RequestID)      // attach a unique ID to every request
r.Use(chimw.RealIP)         // normalize client IP from X-Forwarded-For
r.Use(chimw.Recoverer)      // catch panics, return 500 instead of crashing
r.Use(slogRequestLogger)    // log method, path, status, duration, request ID
r.Use(chimw.Timeout(30s))   // kill any request taking more than 30s
r.Use(cors.Handler(...))    // allow the frontend origin to call us

That chain runs on every request. Then groups of routes add their own middleware on top:

  • Public auth routes (/api/auth/*): add rate-limit middleware with scope "auth".
  • Session-protected routes (/api/* except auth): add RequireProviderSession which enforces the login cookie.
  • Paywalled routes (a subset of /api/*): additionally require an active Stripe subscription.
  • Portal routes (/portal/*): require a valid magic-link token in the URL, not a cookie.
  • Stripe webhook (/webhooks/stripe): bypasses auth entirely because Stripe signs the payload itself.

Why no frameworks? Go discourages Rails-style "here's your whole app in a box." The stdlib's net/http is production-ready, and chi just fills in URL params and sub-routers. If you've used Express.js, chi will feel instantly familiar.

5. Authentication (two separate systems)

This is the part most backends get wrong, so it's worth understanding clearly. There are two completely separate auth systems, used for different people:

5a. Session cookies (for providers who log in)

The daycare owner who runs the center logs in and comes back daily. For them we use a classic session cookie:

  1. User enters email → POST /api/auth/signin.
  2. Backend generates a magic-link token, emails them a link like https://app.example.com/auth/callback?t=abc123.
  3. User clicks link → GET /api/auth/callback consumes the token, creates a row in the sessions table, and sets an HttpOnly Secure cookie named ck_sess with the session token.
  4. Every subsequent request includes that cookie automatically. The RequireProviderSession middleware reads the cookie, looks up the session, and puts provider_id and user_id on the request context.
  5. Handlers read those IDs with middleware.ProviderIDFrom(ctx).

The cookie is HttpOnly (JavaScript can't read it — protects against XSS) and Secure in production (only sent over HTTPS). There are no passwords anywhere — we never have to hash, rotate, or reset them.

5b. Magic links (for parents, staff, signers)

A parent uploading their child's immunization record doesn't have an account. A staff member submitting their CPR certificate doesn't either. Neither does someone asked to sign a policy document. For all of them we use signed, single-purpose, time-bound links.

Look at internal/magiclink/magiclink.go. Key properties:

  • Tokens are random (base62 IDs) — not guessable.
  • Only a hash is stored in the DB (HMAC-SHA256 with a 32+ byte server secret). If someone dumps the database, the links themselves are not recoverable.
  • Every token has a Kind — signup, signin, parent-upload, staff-upload, document-sign. Middleware checks that the kind matches the route, so a signup link can't be used to upload a file.
  • Every token has an expiry. Signup/signin: 15 minutes. Parent/staff upload: 7 days (sliding — extends on each use). Document sign: 72 hours.
  • Single-use kinds are stamped consumed_at on first use. Reusing the link returns "already consumed."

The RequireIndividualMagicLink middleware reads the token from ?t=… or Authorization: Bearer …, calls magic.Consume, and attaches the resulting Claim (token kind, provider ID, subject ID) to the request context.

5c. The billing gate

A third middleware, RequireStripeCustomer, runs after RequireProviderSession on paywalled routes. It asks the billing service "does this provider have an active subscription?" and returns 402 Payment Required if not. This is how PDF signing and (eventually) other premium features are gated.

Why this split matters: the provider's session never goes near a parent's browser, and a parent's upload link never gives them access to the provider's data. Each identity is scoped to exactly what it needs.

6. Database & migrations

The database itself

You're on SQLite. That surprises people. Here's why it's the right choice for ComplianceKit MVP:

  • Zero operational burden. No separate DB server to provision, patch, back up, or restart. It's a file.
  • Stupidly fast for read-heavy workloads. Dashboards and reports fly.
  • WAL mode + busy_timeout (set in db.Open) gives you concurrent readers and a single writer with automatic retry — enough for thousands of providers.
  • Backups are cp ck.db ck.db.bak (or S3 sync, or Litestream).

The pragmas set on every connection:

journal_mode = WAL         # write-ahead log; readers don't block writers
synchronous  = NORMAL      # fast and still safe under WAL
foreign_keys = ON          # SQLite leaves FK enforcement OFF by default (!)
busy_timeout = 5000        # wait up to 5s on writer contention
temp_store   = MEMORY      # keep temp btrees out of the filesystem

The pool is tuned for SQLite's single-writer model — 8 open, 4 idle. Beyond that you'd just manufacture SQLITE_BUSY errors.

There's a db.Tx(ctx, pool, func(tx) error {...}) helper: if your callback returns an error (or panics), the transaction rolls back. Otherwise it commits. Use this whenever you do more than one write.

Heads up — driver mismatch: the db package uses modernc.org/sqlite (pure Go, no cgo) and returns *sql.DB. Some older code (like internal/magiclink) still references pgxpool.Pool from the Postgres days. That's a known inconsistency from the SQLite migration — either the magiclink package needs porting to database/sql, or the project needs to settle on one driver across all packages. Worth a grep for pgxpool before your next deploy.

Migrations

Schema changes live in backend/migrations/ as numbered SQL files. Today you have 9:

000001_init_providers_and_users
000002_children
000003_staff
000004_documents
000005_pdfsign
000006_compliance_and_notifications
000007_billing_and_legal
000008_audit_and_activity
000009_sessions_chase_drills

Each migration has two files: NNNNNN_name.up.sql (apply) and NNNNNN_name.down.sql (roll back). The tool that runs them is golang-migrate/migrate.

Commands you'll actually use

make migrate-up                    # apply all pending migrations
make migrate-down                  # roll back one migration (dev only!)
make migrate-new name=add_foo_bar  # create a new pair of empty .up/.down files
make migrate-version               # show current schema version

The rules (from migrations/README.md)

  1. Sequential integers, never timestamps. Merge conflicts are visible in PR review.
  2. Both up and down are mandatory. A migration with an untested rollback is rejected.
  3. One logical change per file. If "and" appears in the filename, split it.
  4. In production we roll forward, not back. Bad migration? Write a new migration that undoes it. Running migrate down against prod is a paging event.
  5. For live schema changes: add columns as NULL, backfill in a separate script, create indexes with CONCURRENTLY (Postgres), rename in three steps (add-new → dual-write → drop-old).

7. Logging & observability

Logging uses Go's built-in log/slog — structured logging, JSON output, levels (debug/info/warn/error).

Every log line carries a set of attributes. The defaults (set in config.NewLogger) are app and env. Each subsystem adds its own component attribute — look at how handlers are built:

children := &handlers.ChildHandler{Pool: pool, Log: log.With("component", "children")}

Now every log from the children handler includes component=children. You can filter logs by component in whatever log viewer you end up using (CloudWatch, Datadog, jq).

The request logger

slogRequestLogger in router.go emits one line per HTTP request with:

method   path      status  bytes  took       request_id   remote
GET      /api/me   200     412    2.1ms      abc123xyz    73.12.x.x

request_id is the one set by chimw.RequestID — it's also visible in response headers, so if a user reports a bad request you can ask for the request ID and grep your logs.

What you get for free

  • chimw.Recoverer turns panics into 500 responses and logs a stack trace instead of crashing the whole process.
  • All log output goes to stdout as JSON — the right shape for any modern log collector.
  • Log level is env-driven: LOG_LEVEL=debug for local, info for prod.
You don't have metrics or tracing yet. That's fine for now. If you need them later, the wiring is easy: add Prometheus /metrics with promhttp.Handler, or add OpenTelemetry middleware. Don't bother until you have a real performance question.

8. Rate limiting

Simple in-memory token bucket in internal/middleware/ratelimit.go. Each (scope + client IP) gets a bucket with capacity 10 and a refill rate of 0.5 tokens/sec — so in burst you can make 10 quick requests, but sustained you're at ~1 every 2 seconds. When the bucket is empty the middleware returns 429 Too Many Requests with a Retry-After header.

Scopes currently in use: "auth" on signup/signin (stops brute force email enumeration), "portal" on the parent/staff upload endpoints.

It's in-memory — if you ever horizontally scale to multiple instances you'll want Redis-backed rate limiting instead. Not an MVP concern.

9. External services

ServicePackageWhat it's for
AWS S3internal/storageFour buckets: documents (user files), signed-pdfs (finished signed docs), audit-trail (immutable audit logs), raw-uploads (temp holding for unprocessed files).
AWS SESinternal/notifyTransactional email: magic links, deadline alerts.
Stripeinternal/billingSubscriptions, checkout, customer portal, webhooks.
Twiliointernal/notifySMS for deadline nudges.
Mistral + Geminiinternal/ocrOCR with a fallback chain: Mistral first, Gemini if Mistral's confidence is below 0.6. Gemini also runs an "expiration date extractor" on processed text.

All external clients are constructed once at startup and passed into handlers as struct fields. They're safe for concurrent use. Handlers never construct their own clients — this makes testing and mocking easy later.

The S3 upload flow (worth understanding)

  1. Provider hits POST /api/documents/presign with file name and type.
  2. Backend generates a pre-signed URL — a time-limited signed URL the browser can PUT directly to.
  3. Browser uploads the file straight to S3. The bytes never touch your server. This keeps your backend cheap and fast.
  4. Once uploaded, the browser calls POST /api/documents/{id}/finalize.
  5. The finalize handler records the document row, kicks off OCR, and extracts expiration dates.

The Stripe webhook (special case)

POST /webhooks/stripe bypasses all auth middleware — Stripe doesn't have your session cookie. Instead, Stripe signs the raw request body with a secret you share, and the handler verifies that signature. You must use the raw body — any JSON parsing before signature verification breaks the signature. That's why the webhook is mounted at the top level, outside the /api group.

10. Background work (the chase service)

One daily background job runs inside the same process:

chase := notify.NewChaseService(...)
go chase.RunDaily(rootCtx)

go X() launches X in a goroutine — a lightweight thread the Go runtime schedules. RunDaily loops forever sleeping until the next run time, then queries the DB for items that need chasing (expiring documents, overdue uploads, missed drills), and sends emails/SMS via SES and Twilio.

It stops when rootCtx is cancelled (i.e. on Ctrl-C or SIGTERM), which is the same mechanism that stops the HTTP server.

When to break this out: in-process background work is fine until you need retries, schedules from admins, or redundancy across machines. Then you move to something like River, Asynq, or a cron + dedicated worker binary. Not today.

Generating the signing key:

Put that in MAGIC_LINK_SIGNING_KEY. The validator refuses anything shorter than 32 bytes.

12. Running it locally & deploying

Local

# 1. install the migrate CLI once
brew install golang-migrate

# 2. copy .env.example → .env (if you don't have one, see config.go for the full list)

# 3. apply migrations
make migrate-up

# 4. run the server
make run

# 5. confirm it's alive
curl http://localhost:8080/healthz
# → ok

Build the binary

make build       # produces ./bin/compliancekit-api (static, no CGO)

The binary is fully static. You can scp it to any Linux box and run it — no runtime to install.

Docker

make docker-build         # build the image
docker compose up         # start server + any bundled deps locally

Production shape (suggested)

  • One VM (or Fly.io machine, or Render service) running the binary.
  • SQLite file on an attached persistent volume.
  • Litestream replicating the SQLite file to S3 every few seconds (cheap, bulletproof backups).
  • Caddy or an upstream load balancer terminating HTTPS in front.
  • Env vars supplied by the host (never checked in).

13. What you actually need to do

Your mental checklist when touching the backend

  1. Adding a feature? Create or pick a package under internal/, write a handler struct with dependencies as fields, add routes in internal/api/router.go, and pick the right middleware group (public / session / paywalled / magic-link).
  2. Adding a DB column or table? Run make migrate-new name=add_thing, edit both .up.sql and .down.sql, test locally with migrate-up and migrate-down, commit both files together.
  3. Adding config? Add the field to internal/config/config.go. Mark it ,required only if the server cannot function without it.
  4. Adding a log line? Use the handler's h.Log, not slog.Default() — you'll get the component attribute automatically.
  5. Calling an external API? Wire the client in main.go once at startup and pass it into the handler struct. Never construct clients per-request.
  6. Before shipping: make fmt, make test, make build. If all three pass, you're clear. Commit and push — that's the house rule.
